## 2. 从理论上看，今天做了什么

### 2.1 明确了当前工作的理论定位

当前工作的理论定位已经比较清楚：

- 原始项目对应的是论文 *Fast Zone-Based Algorithms for Reachability in Pushdown Timed Automata* 中的 PDTA zone-based reachability 框架；
- 我现在做的是在这个已有框架上，尝试把原本主要处理非对角线约束的 symbolic reachability，扩展到支持**对角线约束**；
- 支撑这个扩展的核心思路，不再是单独依赖传统 LU-extrapolation，而是采用：

```text
LU(G(q)) + diagonal implication
```

也就是说，对状态 `q`：

- 从 `G(q)` 中抽出 simple constraints，构成本地 `LU(G(q))`；
- 从 `G(q)` 中抽出 diagonal constraints，构成 `G^+(q)`；
- subsumption / equivalence 判定时，先做 `LU(G(q))` 部分，再做 diagonal implication。

这和 `todo.tex` 里的理论主线是一致的。

### 2.2 把理论对象和实现对象的对应关系补清楚了

今天最重要的理论推进，不是再发明新算法，而是把“理论里说的对象”和“代码里实际存在的结构”一一对应起来。

目前已经明确：

- 理论中的 `G(q)`：
  - 由当前控制点对应 location 的 invariant；
  - 加上 outgoing edges 的 guards；
 共同组成。

- 理论中的 `G^+(q)`：
  - 对应代码里缓存摘要中的 `diagonals`；
  - 即所有 diagonal clock constraints。

- 理论中的 `LU(G(q))`：
  - 对应代码里缓存摘要中的 `L/U`；
  - 它们来自 simple constraints 的投影：
    - `x - 0 # c` 投到 `U(x)`
    - `0 - x # c` 投到 `L(x)`

- 理论中的 G-simulation 分解：

```text
non-diagonal part  =  LU(G(q))
diagonal part      =  implication over G^+(q)
```

这意味着当前实现已经不是“LU + 对角线外挂”这样一个很松散的说法，而是可以明确地说：

> 当前实现采用了一个基于状态 guard 集合 `G(q)` 的两段式 subsumption 原型：
> 先用 `LU(G(q))` 处理非对角线部分，再用 diagonal implication 处理对角线部分。

### 2.3 有限性部分进一步从“文档证明”走向“实现结构”

`todo.tex` 里原本已经整理了 finite simulation / strong finiteness / Boolean abstraction 的证明路线。

今天做的推进是：

- 虽然没有显式实现一个单独的 Boolean abstraction 类；
- 但是已经把与 finite 相关的“状态摘要”结构工程化出来了。

也就是说，当前实现已经有了一个可缓存的状态摘要，它把：

- `G^+(q)`；
- `LU(G(q))`

统一放在一个实现对象里。

这一步的意义在于：

- 以前 finite 只是文档中的证明思路；
- 现在 finite 至少已经有了**工程上的承载结构**；
- 后续如果要继续做 Boolean abstraction 或更严格的 finite/strongly-finite 工程化，就有自然的落点，不需要推翻现有代码。

## 3. 从实现上看，今天做了什么

### 3.1 补充了代码注释和理论映射

今天在 `include/tchecker/zg/details/zg.hh` 中补充了注释，明确说明：

- `guard_summary_t` 是当前 G-simulation 原型中的状态摘要；
- `guard_summary.diagonals` 对应 `G^+(q)`；
- `guard_summary.L/U` 对应当前实现中的 `LU(G(q))`；
- `guard_summary(...)` / `build_guard_summary(...)` 的作用，是为当前 symbolic control state 构建并缓存这个摘要。

这一步没有改变算法行为，但极大增强了“可解释性”和“可汇报性”。

### 3.2 更新了理论文档 `todo.tex`

今天在 `todo.tex` 中新增了“Current implementation correspondence” 小节，核心意思是：

- 当前代码已经按 `LU(G(q)) + diagonal implication` 结构组织；
- 当前缓存摘要就是理论对象在实现中的对应；
- 仍未显式实现的 Boolean abstraction 作为后续工作保留。

这样之后，`todo.tex` 不再只是“理想中的证明草稿”，而是开始反映当前实现到底做到哪一步。

### 3.3 新增了一个状态相关 `LU(G(q))` 的最小例子

新增例子：

- `MyExamples/state_local_lu.txt`

它的设计目标不是展示 diagonal 本身，而是展示：

- 同一个控制点 `q`；
- 不同整数变量估值 `n`；
- 会导致 `G(q)` 中的 simple constraints 不同；
- 从而导致本地 `LU(G(q))` 不同。

例子的关键 guard 是：

```text
x >= 3 && x <= n
```

两条路径分别让：

- 一条路径设 `n = 1`
- 一条路径设 `n = 5`

于是：

- `n = 1` 分支无法到达最终错误位置；
- `n = 5` 分支可以到达最终错误位置。

这个例子把“`G(q)` 依赖于状态（特别是整数变量估值）”这件事固定成了一个可复现模型。

### 5.2 再演示对角线 TPDA：可达 / 不可达对照

运行：

```sh
./build/src/tchecker -pdta sim MyExamples/ex_tpda.txt
./build/src/tchecker -pdta sim MyExamples/ex_tpda_structural.txt
```

讲法：

- `ex_tpda.txt`：保留对角线约束，最终状态 `q7` 可达；
- `ex_tpda_structural.txt`：不靠矛盾 guard，而是靠栈结构阻断最终状态，因此 `q7` 不可达；
- 这说明扩展后的系统能够处理“对角线 + push/pop”的组合。

### 5.3 再演示无限 push/pop + 对角线：可达 / 不可达对照

运行：

```sh
./build/src/tchecker -pdta sim MyExamples/loop_push_pop.txt
./build/src/tchecker -pdta sim MyExamples/loop_push_pop_match.txt
```

讲法：

- 两个例子都有无限 push/pop 风格结构；
- 都包含时间限制和对角线约束；
- 一个由于 push/pop 次数不匹配导致最终状态不可达；
- 一个由于时间窗允许匹配导致最终状态可达。

这一步适合强调：

- 当前原型不仅能处理简单 TA 式 diagonal guard；
- 也能处理和栈行为、时间窗口结合的 diagonal TPDA 场景。

运行：
